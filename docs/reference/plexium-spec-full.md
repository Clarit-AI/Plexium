# Plexium: Self-Documenting Repositories via Karpathy's LLM Wiki Pattern

## Product Requirements Plan (PRP) — Final Specification

---

## Executive Summary

Plexium transforms repositories into self-documenting systems by applying Karpathy's [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) pattern to agentic coding workflows. Instead of stateless RAG rediscovery on every session, LLM coding agents incrementally build and maintain a persistent, interlinked wiki — a compiled knowledge layer that compounds with every commit, every conversation, and every ingested source.

The codebase is the raw source layer. The `.wiki/` vault is the synthesized knowledge layer. The schema is the governance layer. Agents read the wiki before working and update it after every change. Git hooks and CI pipelines enforce this discipline. The wiki is simultaneously an Obsidian vault, a GitHub Wiki (via submodule), and the canonical context surface for every coding assistant — Claude Code, OpenAI Codex, Gemini CLI, Cursor, and any future agent.

**The pitch:** *Give it your repo, and it builds, maintains, and enforces a living wiki that makes every agent session smarter than the last.*

## 1. The Problem

Repositories accumulate code but shed context. Documentation goes stale because the maintenance burden grows faster than the value. Meanwhile:

- **Agents suffer from Groundhog Day.** Every session re-derives understanding from scratch. Claude Code reads your codebase anew each time. Codex re-discovers architecture on every task. Nothing compounds.
- **Single-file solutions collapse at scale.** `CLAUDE.md` and `.cursorrules` work for prototypes. A 100,000-line system cannot fit its architecture, decisions, patterns, contradictions, and module responsibilities into a single prompt file.
- **GitHub Wikis are scriptable but abandoned.** They are separate Git repos (`REPO.wiki.git`) — fully programmable — but nobody maintains them because the bookkeeping is tedious.
- **RAG is stateless.** Upload docs, retrieve chunks, generate answers. The LLM pieces together fragments on every query. No synthesis accumulates. No contradictions are flagged. No cross-references are built.

**What's missing from every project is the middle layer:** a persistent, cross-referenced, LLM-maintained synthesis of what the code *means*, why decisions were made, how subsystems interact, and where the contradictions lie. Karpathy's LLM Wiki pattern provides this layer. Plexium operationalizes it for software repositories with mandatory enforcement.

## 3. Vault Structure

The wiki lives at **`.wiki/`** at the repository root. This placement mirrors `.github/`, `.vscode/`, and `.husky/` — a dotfolder that signals tooling infrastructure, keeping the project root clean while remaining immediately discoverable.

`.wiki/` is simultaneously:
- An **Obsidian vault** (with `.obsidian/` config inside)
- A **git submodule** pointing to `https://github.com/{org}/{repo}.wiki.git`, natively powering the GitHub Wiki tab
- The **canonical knowledge layer** for all coding agents

## 4. The Universal Schema

`_schema.md` is the constitution. It transforms a generic LLM into a disciplined wiki maintainer. Written once, referenced by every agent-specific instruction file via adapter plugins.

```markdown
# PLEXIUM SCHEMA v1 — MANDATORY AGENT DIRECTIVES

You are the custodian of the `.wiki/` vault in this repository. Your memory
does not persist between sessions, but this vault does. It is the compiled,
persistent knowledge of this entire codebase.

## MANDATORY WORKFLOW — EVERY TASK

### 1. READ (before any code change)
- Read `.wiki/_index.md` to orient yourself.
- Fetch relevant module, architecture, and decision pages for your work area.
- If a retrieval tool is available (PageIndex MCP, `plexium retrieve`),
  use it instead of scanning files manually.
- Check `.wiki/_log.md` (last 10 entries) for recent context.
- Check page `ownership` frontmatter before modifying any wiki page.

### 2. EXECUTE
- Perform the coding task requested by the user.

### 3. DOCUMENT (FORBIDDEN to end your task without this step)
- Update every `.wiki/modules/*.md` page affected by your changes.
- If you made an architectural decision, create or update a `.wiki/decisions/*.md` ADR.
- If you discovered a contradiction, add it to `.wiki/contradictions.md`.
- Add an entry to `.wiki/_log.md` (see LOG FORMAT below).
- Update `.wiki/_index.md` if you created or removed pages.
- Update cross-references ([[wiki-links]]) on pages whose relationships changed.
- NEVER modify pages with `ownership: human-authored` unless explicitly instructed.
- For `ownership: co-maintained` pages, append only — do not rewrite existing sections
  unless the user specifically requests it.

### 4. VALIDATE
- Confirm wiki updates are consistent with the code you actually wrote.
- Mark uncertain claims with `<!-- CONFIDENCE: low — needs human review -->`.
- Verify all [[wiki-links]] you created resolve to existing pages.
- Verify `source-files` frontmatter references existing paths.

## TRIVIAL CHANGE EXCEPTION
For changes affecting only a single file with no architectural impact
(typo fixes, version bumps, formatting): a brief `_log.md` entry suffices.
Full wiki update not required.

## LOG FORMAT
Each entry in `_log.md` must use this parseable format:

  ## [YYYY-MM-DD] {task|ingest|lint|query|convert} | Brief description
  - Changed: modules/auth.md, architecture/overview.md
  - Decision: decisions/015-jwt-rotation.md (new)
  - Contradictions: None found
  - Files touched: src/auth/middleware.ts, src/auth/jwt.ts

## PAGE GENERATION RULES

### Slug rules
- Page names must be filesystem-safe (no spaces — use hyphens).
- Duplicate titles must be deduplicated predictably (append qualifier).
- Heading-derived slugs must remain stable across regenerations.

### Navigation rules
- Every generated page must be reachable from _index.md directly or indirectly.
- _Sidebar.md must expose top-level sections and key pages.
- Navigation ordering must be deterministic (alphabetical within sections).

### Content rules
- Preserve factual meaning from source docs and code.
- NEVER invent implementation details not present in sources.
- Summarize when needed but do not silently discard major sections.
- Prefer cross-links ([[wiki-links]]) over duplicated paragraphs.
- Every page must begin with YAML frontmatter (see FRONTMATTER SPEC).

### Cross-reference rules
- When mentioning a concept, module, or decision that has its own page, use [[wiki-links]].
- Never remove existing cross-references without logging the removal in _log.md.
- When creating a new page, add inbound links from at least 2 related existing pages.

## FRONTMATTER SPEC
Every wiki page must begin with:
```yaml
---
title: <Human-readable title>
ownership: managed              # managed | human-authored | co-maintained
last-updated: YYYY-MM-DD
updated-by: <agent-name>
related-modules: [<list>]
source-files: [<glob patterns>]
confidence: high                # high | medium | low
review-status: unreviewed       # unreviewed | human-verified | stale
tags: [<list>]
---
```

## LINT PROTOCOL
When asked to lint, check for:
- Pages not updated in >30 days that reference frequently-changed code
- Orphan pages (no inbound [[links]])
- Concepts mentioned in 3+ pages without their own page
- Contradictions between module pages and architecture overview
- `source-files` in frontmatter referencing paths that no longer exist
- Missing cross-references between related modules
- Pages with `confidence: low` that need investigation
- Managed pages whose source file hashes differ from the state manifest

## INGEST PROTOCOL
When a new raw source is added (meeting note, ticket export, memento transcript):
1. Read it fully
2. Discuss key takeaways with the user (unless batch mode)
3. Write a summary page or update existing pages
4. Update _index.md, _log.md, _Sidebar.md
5. Cross-reference with existing module/decision pages
6. Flag contradictions with existing wiki content
7. Update the state manifest with new source mappings
```

This schema is injected into agent-specific instruction files by the installer:

| Agent | Instruction File | Method |
|-------|-----------------|--------|
| Claude Code | `CLAUDE.md` | Includes full schema excerpt + path references |
| OpenAI Codex | `AGENTS.md` | Same content, Codex format |
| Gemini CLI | `.gemini/config.md` | Same content |
| Cursor | `.cursor/rules/plexium.mdc` | Same content, MDC format |
| OpenCode | `opencode.json` agents config | Registers schema as system context |
| Continue.dev | `.continue/rules/plexium.md` | Same content |
| Future agents | `plexium plugin add <name>` | Plugin writes new instruction file |
| Symphony | `WORKFLOW.md` (injected section) | Plexium directives appended to orchestration contract |

**The plugin model for extensibility:** each agent adapter is a small script in `.plexium/plugins/` that knows where its instruction file lives and how to format the schema injection. Adding support for a new assistant requires only a new plugin — no installer rewrite.

## 5. Page Ownership Model

The single most critical trust mechanism. Every wiki page has an explicit owner.

| Ownership | Meaning | Agent Behavior |
|-----------|---------|----------------|
| `managed` | Created and controlled by Plexium automation. Safe to rewrite. | Agents may freely update, rewrite, or regenerate. |
| `human-authored` | Written by a human. Agents must not modify. | Agents must NEVER overwrite. They may suggest changes via `_log.md` or PR comments. |
| `co-maintained` | Collaboratively maintained. Agents may append, humans structure. | Agents append new sections or update data. Never rewrite existing prose without explicit instruction. |

**Enforcement:**
- Ownership is declared in page frontmatter (`ownership: managed`)
- Ownership is tracked in `.plexium/manifest.json` (machine-readable, deterministic)
- The deterministic lint layer rejects commits where an agent modified a `human-authored` page
- The schema explicitly instructs agents to check ownership before writing
- Optional: a generated marker comment at the top of managed pages:
  ```markdown
  <!-- PLEXIUM:MANAGED — This page is maintained by Plexium. Manual edits may be overwritten. -->
  ```

**Default behavior:**
- Pages created by `plexium init`, `plexium convert`, or agents following the schema default to `managed`
- Pages created manually by humans (detected by absence from manifest) default to `human-authored`
- Users can promote any page to `co-maintained` when they want to add human context while keeping agent updates flowing

### Page Lifecycle State Machine

Beyond ownership, every wiki page has a lifecycle state with explicit transition rules:

```
stub → generated → unreviewed → human-verified → stale → regenerated → unreviewed
```

| State | Meaning | Transition To | Triggered By |
|-------|---------|---------------|-------------|
| `stub` | Placeholder created during convert/init | `generated` | Agent or `plexium sync` fills content |
| `generated` | Content created by automation | `unreviewed` | Automatic on creation |
| `unreviewed` | Content exists but not human-reviewed | `human-verified` | Human PR review approval |
| `human-verified` | Human has confirmed accuracy | `stale` | Source file hashes drift from manifest |
| `stale` | Source has changed since last wiki update | `regenerated` | `plexium sync`, daemon, or agent |
| `regenerated` | Updated after staleness detected | `unreviewed` | Automatic (re-enters review cycle) |

**Transition rules:**
- Only automation (agents, `plexium convert`, `plexium sync`) can move `stub → generated`
- Only humans can move `unreviewed → human-verified` (via PR review)
- `human-verified → stale` is automatic (deterministic hash comparison)
- The daemon (Phase 3) watches for pages entering `stale` and dispatches agents to regenerate

The lifecycle state is tracked in `review-status` frontmatter and in `manifest.json`.

## 6. Page Generation Rules

Concrete, testable rules that keep the wiki structurally sound.

### Slug Rules
- Page names must be filesystem-safe: lowercase, hyphens instead of spaces, no special characters
- Duplicate titles must be deduplicated predictably: append parent directory or qualifier (`auth` vs `auth-middleware`)
- Heading-derived slugs must remain stable — once a slug is assigned, it does not change unless the page is intentionally renamed (tracked in manifest)

### Title Rules
- Prefer human-readable titles over raw filenames
- Avoid raw filename leakage when a better title exists in the content
- Append qualifiers only when necessary to avoid collisions

### Content Rules
- Preserve factual meaning from source docs and code
- Never invent implementation details not present in sources
- Summarize when needed but never silently discard major sections
- Prefer cross-links (`[[wiki-links]]`) over duplicated paragraphs
- Each page should be self-contained enough to be useful without reading all linked pages

### Navigation Rules
- Every generated page must be reachable from `Home.md` / `_index.md` directly or via one intermediate link
- `_Sidebar.md` must expose all top-level sections and high-traffic pages
- Navigation ordering must be deterministic (alphabetical within sections, chronological for decisions/log)
- `_Footer.md` includes last-updated timestamp and link back to `Home.md`

### Page Taxonomy

| Source | Wiki Output | Section |
|--------|------------|---------|
| `README.md` | `Home.md` | Root |
| `src/{module}/` | `modules/{module}.md` | Modules |
| `docs/*.md` | Pages by content type | Architecture / Guides / Concepts |
| `docs/{folder}/` | Section index page + child pages | Varies |
| ADR files | `decisions/NNN-title.md` | Decisions |
| Named domain concepts | `concepts/{concept}.md` | Concepts |
| Recurring patterns | `patterns/{pattern}.md` | Patterns |
| Raw ingested sources | `raw/{category}/{source}` + summary in wiki | Raw (immutable) |

## 7. State Manifest & Mapping

`.plexium/manifest.json` tracks the bidirectional relationship between source files and wiki pages. This is the *machine-readable* companion to the *human-readable* `_index.md`.

```json
{
  "version": 1,
  "lastProcessedCommit": "abc123f",
  "lastPublishTimestamp": "2026-04-05T14:30:00Z",
  "pages": [
    {
      "wikiPath": "modules/auth.md",
      "title": "Auth Module",
      "ownership": "managed",
      "section": "modules",
      "sourceFiles": [
        {
          "path": "src/auth/**",
          "hash": "sha256:e3b0c44...",
          "lastProcessedCommit": "abc123f"
        },
        {
          "path": "docs/auth-flow.md",
          "hash": "sha256:d7a8fbb...",
          "lastProcessedCommit": "abc123f"
        }
      ],
      "generatedFrom": ["src/auth/", "docs/auth-flow.md"],
      "lastUpdated": "2026-04-05",
      "updatedBy": "claude-code",
      "inboundLinks": ["architecture/overview.md", "modules/api-gateway.md"],
      "outboundLinks": ["decisions/003-jwt-strategy.md", "concepts/rbac.md"]
    }
  ],
  "unmanagedPages": [
    {
      "wikiPath": "architecture/legacy-notes.md",
      "firstSeen": "2026-03-15",
      "ownership": "human-authored"
    }
  ]
}
```

**The manifest enables:**
- Forward mapping: "Which wiki pages were generated from `src/auth/`?"
- Reverse mapping: "Which source files feed into `modules/auth.md`?"
- Staleness detection: "Has `src/auth/middleware.ts` changed since its wiki page was last updated?" (hash comparison)
- Ownership enforcement: "Is this page managed or human-authored?"
- Idempotency: "Was this commit already processed?" (lastProcessedCommit)
- Link integrity: "Do all inbound/outbound links resolve?"

**Update rules:**
- The manifest is updated by the deterministic pipeline, not by LLM agents directly
- Agents update wiki pages; the `plexium` CLI updates the manifest from the resulting state
- CI validates manifest consistency on every PR

### Assistive Agent Configuration

The assistive agent is an optional lightweight LLM that handles low-complexity wiki tasks at minimal cost. It operates through a provider cascade — try the cheapest viable option first, escalate on failure.

```yaml
# Additional .plexium/config.yml fields

assistiveAgent:
  enabled: true
  
  # Provider cascade (tried in order)
  providers:
    - name: local
      enabled: false                    # Opt-in for users with hardware
      type: ollama
      endpoint: "http://localhost:11434"
      model: "gemma4:26b-a4b"
      
    - name: openrouter-free
      enabled: true                     # Default on
      type: openai-compatible
      endpoint: "https://openrouter.ai/api/v1"
      model: "openrouter/free"          # Auto-routes to best free model
      apiKeyEnv: "OPENROUTER_API_KEY"
      rateLimits:
        requestsPerMinute: 20
        requestsPerDay: 200
      
    - name: openrouter-budget
      enabled: false                    # Opt-in when free tier is too constrained
      type: openai-compatible
      endpoint: "https://openrouter.ai/api/v1"
      model: "google/gemma-4-26b-a4b-it"
      apiKeyEnv: "OPENROUTER_API_KEY"
      maxMonthlySpend: 5.00
      
    - name: primary                     # Always last — the expensive fallback
      enabled: true
      type: inherit                     # Uses whatever the coding agent is
  
  # Task routing
  routing:
    assistive:                          # Route to Tier 1/2 providers
      - frontmatter-update
      - log-entry  
      - index-regeneration
      - sidebar-regeneration
      - link-validation
      - cross-reference-suggestion
      - manifest-update
      - page-state-transition
      - module-summary
      - staleness-check
      - wiki-debt-logging
    primary:                            # Always route to Tier 3
      - architecture-synthesis
      - contradiction-detection
      - adr-creation
      - complex-ingest
      - deep-code-analysis
  
  # Rate limit management
  rateLimits:
    strategy: adaptive                  # adaptive | strict | ignore
    batchSize: 5
    cooldownMs: 3100
    dailyBudget: 180
  
  # Bulk operation fallback
  bulkOperations:
    tier: budget

## 9. The CLI (`plexium`)

A single CLI distributed as a **standalone binary** (Go or Rust, cross-compiled) with a fallback `curl | sh` install. Language-agnostic — does not couple to Node, Python, or any project ecosystem.

### Installation

```bash
# Primary: standalone binary
curl -fsSL https://plexium.dev/install.sh | sh

# Alternatives
brew install plexium
go install github.com/plexium-dev/plexium@latest
```

### Commands

| Command | Description |
|---------|-------------|
| `plexium init [options]` | Scaffold `.wiki/` structure, config, agent adapters, hooks, CI |
| `plexium convert [options]` | Brownfield: scour existing docs/code, bootstrap wiki |
| `plexium sync [options]` | Incremental sync after changes |
| `plexium lint [options]` | Health-check the wiki |
| `plexium bootstrap` | Full regeneration of wiki from sources (destructive to managed pages) |
| `plexium retrieve "<query>"` | Query the wiki via PageIndex or index-based search |
| `plexium publish` | Push wiki to GitHub Wiki repo |
| `plexium gh-wiki-sync` | Selective sync to GitHub Wiki with publish/exclude filtering |
| `plexium doctor` | Validate config, manifest, hooks, CI, agent adapters |
| `plexium migrate` | Apply schema migrations across all existing pages |
| `plexium plugin add <name>` | Add adapter plugin for a new agent |
| `plexium hook pre-commit` | Run pre-commit validation (called by Lefthook) |
| `plexium ci check` | Run CI validation (called by GitHub Actions) |
| `plexium daemon [options]` | Start autonomous wiki maintenance loop (opt-in, Phase 3) |

| `plexium orchestrate --issue <ID>` | Run a single orchestrated wiki-update for one issue |

### `plexium init`

```bash
plexium init [options]
```

| Flag | Effect |
|------|--------|
| `--agents claude,codex,gemini,cursor` | Select agent configs (auto-detects by default) |
| `--strictness strict\|moderate\|advisory` | Hook enforcement level (default: `moderate`) |
| `--with-memento` | Initialize memento integration |
| `--with-beads` | Initialize beads task graph integration |
| `--with-pageindex` | Set up PageIndex MCP server for retrieval |
| `--obsidian` | Include `.obsidian/` config with recommended plugins |
| `--github-wiki` | Initialize `.wiki/` as submodule of `{repo}.wiki.git` |
| `--private` | Add sensitivity rules, exclude `raw/` from publish |
| `--dry-run` | Preview what would be created without writing files |
| `--assistive-agent local,openrouter-free` | Configure assistive agent providers (default: `openrouter-free`) |

**What `init` does:**
1. Creates `.wiki/` directory structure with template pages
2. Creates `.plexium/` with `config.yml`, empty `manifest.json`, plugin scripts
3. Writes `_schema.md` customized for the detected tech stack
4. Generates agent-specific instruction files via adapter plugins
5. Installs git hooks via [Lefthook](https://github.com/evilmartians/lefthook)
6. Adds `.github/workflows/plexium-lint.yml` and `plexium-sync.yml`
7. If `--github-wiki`: initializes git submodule for `{repo}.wiki.git`
8. If `--obsidian`: writes `.obsidian/` config
9. If `--with-memento`: runs `git memento init`, adds memento CI gate
10. If `--with-beads`: scaffolds beads integration config
11. If `--with-pageindex`: sets up PageIndex MCP server config
12. Creates initial `_index.md`, `_log.md`, `Home.md`, `_Sidebar.md`
13. Initializes `manifest.json` with empty state
14. Emits a bootstrap report to `.plexium/reports/`

### `plexium convert`

```bash
plexium convert [--depth shallow|deep] [--agent claude|codex|gemini] [--dry-run]
```

Multi-phase brownfield ingestion:

| Phase | What Happens |
|-------|-------------|
| **Scour** | Traverses directory tree. Reads README files, JSDoc/docstrings, inline comments, `package.json`/`Cargo.toml`/`pyproject.toml`, CI configs, existing ADRs, OpenAPI specs, `.env.example`. Scans git history for major milestones. Reads existing `CLAUDE.md`/`AGENTS.md`/`.cursorrules`. |
| **Filter** | Classifies eligible files. Applies include/exclude globs from config. Normalizes markdown structure. |
| **Ingest** | Translates raw findings into structured `.wiki/` format. Creates module pages from directory structure. Extracts decisions from commit messages and PR descriptions. Builds architecture overview from import graphs and configuration. |
| **Link** | Generates cross-references between pages. Builds `_index.md`, `Home.md`, `_Sidebar.md`. |
| **Lint** | Gap analysis. Creates stub pages labeled `<!-- STATUS: stub — needs deeper analysis -->` for undocumented modules. Highlights missing documentation. |
| **Report** | Generates `_conversion-report.md` showing what was captured, inferred, and missing. Emits structured report to `.plexium/reports/`. |

**Dry-run mode:** `plexium convert --dry-run` generates all output to `.plexium/output/` without modifying `.wiki/` or committing anything. Essential for building trust during initial brownfield adoption.

## 10. Workflows & Operations

### Operation 1: Bootstrap (First Run)

**Goal:** Convert an existing repo's documentation into an initial wiki.

**Trigger:** `plexium init` + `plexium convert` (manual).

**Inputs:**
- Repo root contents
- README.md, `/docs/**`, ADRs, design notes
- Configuration from `.plexium/config.yml`

**Optional inputs:**
- Include/exclude globs
- Preferred page taxonomy overrides
- ADR directory override
- Max page size threshold
- Tone/style preferences

**Process:**
1. Scan source docs
2. Filter eligible files
3. Normalize markdown structure
4. Classify pages into sections
5. Generate initial wiki pages
6. Generate `Home.md`
7. Generate `_Sidebar.md` and `_Footer.md`
8. Write mapping/state to `manifest.json`
9. Publish to `.wiki/` (and optionally to wiki repo)
10. Emit bootstrap report

**Outputs:**
- `Home.md`, `_Sidebar.md`, `_Footer.md`
- Section pages, module pages, decision pages, concept pages
- `_index.md` (master catalog)
- `_log.md` (initial entry)
- Optional: `open-questions.md`, `contradictions.md`
- `manifest.json` (fully populated)
- Bootstrap report (Markdown + JSON)

### Operation 2: Incremental Sync (After Changes)

**Goal:** Keep the wiki current after code or documentation changes.

**Trigger:**
- Automatic: agent follows schema during coding session
- Manual: `plexium sync`
- CI: merge to main triggers sync workflow
- Label: PR with `wiki-sync` label

**Inputs:**
- Changed files list or git diff
- `manifest.json` (current state)
- `.plexium/config.yml`

**Process:**
1. Inspect changed files
2. Determine wiki-relevance (source files mapped in manifest?)
3. Resolve impacted wiki pages
4. Regenerate or update affected pages
5. Refresh `_Sidebar.md`, `Home.md` if taxonomy changed
6. Update `manifest.json`
7. Publish only changed wiki files
8. Emit sync report

**Idempotency guarantee:** Running sync twice on the same commit produces identical output. The manifest's `lastProcessedCommit` field enables skip-if-already-processed logic.

### Operation 3: Query

**Goal:** Answer questions against the wiki, optionally filing good answers back.

**Trigger:** User asks agent a question or runs `plexium retrieve "<query>"`.

**Process:**
1. Read `_index.md` for orientation
2. Use retrieval tool (PageIndex MCP, or index-based fallback) to find relevant pages
3. Read relevant pages
4. Synthesize answer with citations (`[[wiki-links]]`)
5. If the answer is reusable (comparison, analysis, synthesis), file it as a new wiki page
6. Update `_index.md` and `_log.md`

**Insight:** Good answers compound in the knowledge base. A comparison table, an architectural analysis, a discovered connection — these shouldn't disappear into chat history.

### Operation 4: Lint / Health Check

**Goal:** Evaluate the structural and semantic health of the wiki.

**Trigger:**
- Manual: `plexium lint`
- Scheduled: weekly GitHub Action
- CI: runs on every PR

**Process — Deterministic checks (always enforced):**
1. Crawl all wiki pages
2. Resolve all `[[wiki-links]]` — report broken links
3. Detect orphan pages (no inbound links)
4. Detect stale pages (source files changed since `last-updated`)
5. Validate `source-files` frontmatter references existing paths
6. Validate `manifest.json` consistency with actual wiki state
7. Verify `_Sidebar.md` links resolve
8. Flag pages with `confidence: low`

**Process — LLM-augmented checks (advisory):**
1. Detect contradictions between module pages and architecture overview
2. Identify concepts mentioned in 3+ pages without their own page
3. Suggest missing cross-references
4. Propose new pages for undocumented areas
5. Detect semantic staleness (content is technically present but outdated)

**Outputs:**
- Lint report (Markdown + JSON)
- Severity levels: `error` (broken links), `warning` (stale pages), `info` (suggestions)

**Invariant:** Lint mode NEVER modifies source content or wiki pages. It only reports.

### Operation 5: Ingest

**Goal:** Process new raw source documents into the wiki.

**Trigger:** User drops a document into `.wiki/raw/` and instructs the agent.

**Process:**
1. LLM reads the source fully
2. Discusses key takeaways with the user (unless batch mode)
3. Writes a summary page in the wiki
4. Updates `_index.md`, `_log.md`
5. Updates relevant entity and concept pages across the wiki
6. Appends source mapping to `manifest.json`
7. Flags contradictions with existing content

A single source may touch 10–15 wiki pages.

### Operation 6: Autonomous Maintenance Daemon

**Goal:** Fix documentation drift asynchronously without human supervision.

**Trigger:** `plexium daemon` (long-running process, Phase 3).

**Process:**
1. Poll for triggers: new commits with unmapped wiki changes, stale pages past threshold, WIKI-DEBT entries, new files in `.wiki/raw/`, lint findings above severity threshold
2. Claim eligible work items (bounded by `daemon.maxConcurrent`)
3. Create isolated git worktree for each work item (`.plexium/workspaces/<ID>/`)
4. Sequence agent roles: Retriever (context via PageIndex) → Coder (if code changes needed) → Documenter (wiki updates) → Linter (validation)
5. On success: open PR with Wiki Integrity Report, transition to `Human Review` handoff state
6. On failure: retry with exponential backoff, then release work item and log to `_log.md`
7. After each run: `plexium compile` regenerates shared navigation files

**Concurrency model:** Under parallel daemon runs, `_index.md`, `_Sidebar.md`, and `Home.md` become compiled deterministic outputs rather than direct agent write targets. Agents update module/decision/concept pages and append to per-run log files. The deterministic `plexium compile` step regenerates shared navigation files after each run — no LLM calls required.

**Recovery:** On restart, the daemon reconciles against `manifest.json` + `lastProcessedCommit` — no external database required.

## 11. Enforcement Layers

Four layers of increasing strictness that make wiki maintenance mandatory, not optional.

### Layer 1: Schema Injection (Soft — Agent Compliance)

The `_schema.md` content in every agent instruction file means compliant agents follow the wiki protocol by default. This handles ~80% of cases. Bypassable if the agent is overridden or instruction files are deleted.

### Layer 2: Git Hooks (Medium — Developer Friction)

Installed via [Lefthook](https://github.com/evilmartians/lefthook) in `lefthook.yml`:

```yaml
pre-commit:
  commands:
    plexium-check:
      run: plexium hook pre-commit
      fail_text: |
        ⚠️  Code files changed but .wiki/ was not updated.
        Ask your coding agent to document the changes, or run:
          plexium sync
        To bypass (with audit trail): git commit --no-verify

post-commit:
  commands:
    plexium-audit:
      run: plexium hook post-commit
```

**`plexium hook pre-commit` logic:**
1. Check `git diff --cached` for changes to source files (per `sources.include` config)
2. If source files changed, verify at least one `.wiki/` file was also staged
3. Behavior by strictness:

| Strictness | Behavior |
|-----------|----------|
| `strict` | Hard rejects the commit. No bypass without `--no-verify`. |
| `moderate` (default) | Warns with clear remediation. Requires explicit `--no-verify` to bypass. |
| `advisory` | Warns but allows the commit. Logs the skip. |

**`plexium hook post-commit` logic:**
- If `--no-verify` was used (detected via absence of pre-commit marker), append a `WIKI-DEBT` entry to `_log.md`:
  ```
  ## [2026-04-05] WIKI-DEBT | Commit abc123 bypassed wiki check
  - Files changed: src/auth/middleware.ts
  - Bypassed by: developer (--no-verify)
  - Status: pending wiki update
  ```
- This creates a trackable backlog of documentation debt

### Layer 3: CI/CD Pipeline (Hard — Merge Blocking)

`.github/workflows/plexium-lint.yml`:

```yaml
name: Plexium Lint

on:
  pull_request:
    branches: [main, develop]

jobs:
  wiki-check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
          submodules: true
          sparse-checkout: |
            .wiki/
            .plexium/
            src/
            docs/
            README.md
          sparse-checkout-cone-mode: false

      - name: Install Plexium
        run: curl -fsSL https://plexium.dev/install.sh | sh

      - name: Deterministic checks
        run: |
          plexium ci check \
            --base ${{ github.event.pull_request.base.sha }} \
            --head ${{ github.event.pull_request.head.sha }} \
            --output .plexium/reports/ci-check.json

      - name: Lint wiki integrity
        run: plexium lint --ci --fail-on error

      - name: Post results to PR
        if: always()
        uses: actions/github-script@v7
        with:
          script: |
            const fs = require('fs');
            const report = JSON.parse(fs.readFileSync('.plexium/reports/ci-check.json'));
            // Post formatted report as PR comment

      - name: Memento gate
        if: hashFiles('.plexium/config.yml') != ''
        run: |
          if grep -q 'memento: true' .plexium/config.yml; then
            git memento check --gate
          fi
```

`.github/workflows/plexium-sync.yml`:

```yaml
name: Plexium Sync

on:
  push:
    branches: [main]
  workflow_dispatch:

jobs:
  sync-and-publish:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
          submodules: true

      - name: Install Plexium
        run: curl -fsSL https://plexium.dev/install.sh | sh

      - name: Sync wiki
        run: plexium sync --ci

      - name: Publish to GitHub Wiki
        env:
          GH_TOKEN: ${{ secrets.WIKI_PUSH_TOKEN }}
        run: plexium gh-wiki-sync --push

      - name: Emit sync report
        run: plexium report --format markdown --output $GITHUB_STEP_SUMMARY
```

**`plexium ci check` performs:**
1. Diffs changed source files against manifest's `sourceFiles` mappings
2. Flags modules with code changes whose wiki pages weren't touched
3. Checks for `WIKI-DEBT` entries accumulated since last CI run
4. Returns non-zero exit if wiki-debt exceeds configured threshold
5. Optionally posts suggested wiki updates as PR review comments (configurable)

### Layer 4: Memento Gate (Provenance)

When memento is enabled, CI fails builds without proper session provenance. Every AI coding session produces a transcript attached as git notes. Transcripts are also copied to `.wiki/raw/memento-transcripts/` and periodically ingested into the wiki — extracting decisions, rationale, and tradeoffs into ADR and module pages. AI reasoning about code decisions becomes persistent project knowledge.

## 12. Deterministic vs. LLM Pipeline

A critical architectural principle: **deterministic checks first, LLM as augmentation.** The deterministic layer is fast, testable, reproducible, and never hallucinates. The LLM layer is powerful but non-deterministic. CI gates depend on deterministic checks. LLM-generated suggestions are advisory until human-verified.

| Operation | Deterministic Layer | LLM Layer |
|-----------|-------------------|-----------|
| Link validation | Crawl pages, resolve all `[[links]]`, report broken | — |
| Orphan detection | Graph traversal from `_index.md`, list unreachable | — |
| Staleness detection | Compare `last-updated` vs source `mtime` / hash | — |
| CI freshness check | Diff source files vs manifest mappings | — |
| Manifest consistency | Validate all paths, hashes, links in manifest | — |
| Sidebar generation | Generate from `_index.md` structure | — |
| Slug deduplication | Deterministic suffix rules | — |
| Page generation (content) | Template structure, frontmatter scaffolding | Content synthesis, cross-referencing |
| Contradiction detection | — | Read multiple pages, flag conflicts |
| Summary generation | — | Synthesize from source |
| Gap analysis | Detect missing pages (filesystem heuristics) | Suggest what pages should exist (semantic) |
| Cross-reference suggestions | — | Identify related concepts across pages |

**The deterministic layer runs without any LLM calls.** This means:
- `plexium lint --deterministic` is fast and free (no API costs)
- CI checks are reproducible — same input always produces same output
- LLM-augmented lint (`plexium lint --full`) adds semantic analysis on top

## 14. GitHub Wiki Integration

Initialize `.wiki/` as a **git submodule** pointing to `https://github.com/{org}/{repo}.wiki.git`:

```bash
plexium init --github-wiki
# Internally runs:
# git submodule add https://github.com/{org}/{repo}.wiki.git .wiki
```

**Dual-purpose publishing:** the wiki files edited locally by agents appear on the GitHub Wiki tab without additional sync tooling. Changes committed to the submodule are immediately visible on GitHub.

**Selective publishing:** the `githubWiki.publish` and `githubWiki.exclude` config arrays control which pages appear on the public wiki tab. Sensitive pages (raw sources, internal contradictions, unreviewed content) can be excluded.

**Navigation files:** `_Sidebar.md` and `_Footer.md` are generated by the deterministic pipeline from `_index.md` structure. GitHub's wiki renderer uses these for navigation chrome. Without them, the wiki tab is a flat, unnavigable list.

**Sync workflow:** on merge to main, the CI workflow:
1. Validates wiki integrity
2. Filters pages per publish/exclude config
3. Commits to the wiki submodule
4. Pushes to `{repo}.wiki.git`

**Auth note:** pushing to `{repo}.wiki.git` requires write access to the wiki repository, which has different permission semantics than the main repo. See [Authentication Model](#17-authentication-model).

## 16. Tool Integrations

### Memento — Session Provenance & CI Gates

[Memento](https://github.com/mandel-macaque/memento) attaches AI coding session transcripts as git notes to commits.

**Integration:**
- `plexium init --with-memento` runs `git memento init`
- Session transcripts are stored as git notes AND copied to `.wiki/raw/memento-transcripts/`
- CI gate (`git memento check --gate`) fails builds without proper session provenance
- The schema instructs agents to periodically ingest transcripts into the wiki, extracting decisions and rationale into ADR and module pages
- This creates a feedback loop: **AI reasoning about code decisions becomes persistent project knowledge**

### Beads — Structured Task Graphs

[Beads](https://github.com/gastownhall/beads) provides dependency-aware task graphs and persistent memory for long-horizon work.

**Integration:**
- The schema instructs agents to use `bd` (beads CLI) for task planning
- As beads graphs resolve, agents synthesize outcomes into wiki decision pages and module updates
- Wiki page frontmatter includes `beads-ids` for bidirectional traceability
- The beads graph captures the *process*; the wiki captures the *knowledge*
- They are mutually reinforcing: beads provides structure for in-flight work, the wiki provides context for starting new work

### PageIndex — Dedicated Retrieval Agent

[PageIndex](https://github.com/VectifyAI/PageIndex) uses hierarchical tree indexing with multi-step reasoning-based search — it navigates the wiki like a human uses a book's table of contents and index, rather than fuzzy embedding similarity. This is superior to vector-based RAG for structured documentation.

**Yes, a dedicated retrieval agent is the proper architecture.** It keeps coding agents focused on their primary task while providing precise, pre-synthesized context.

**Integration model:**

```
Developer asks question
        │
        ▼
Coding Agent (Claude/Codex/Gemini)
        │
        │ "I need context on the auth system"
        ▼
Retrieval Tool
        │
        ├── PageIndex MCP server (if configured)
        │   └── Hierarchical tree search across .wiki/
        ├── OR: `plexium retrieve "auth system"`
        │   └── CLI wrapper around PageIndex
        ├── OR: fallback to _index.md scan
        │   └── Read index, grep for relevant pages
        │
        ▼
Returns: modules/auth.md + decisions/003-jwt-strategy.md
         + relevant sections from architecture/overview.md
        │
        ▼
Coding Agent (now has precise context)
        │
        ▼
Writes code + updates wiki
```

Exposed as:
- **MCP server** — for agents with native MCP support (Claude Code)
- **CLI tool** — `plexium retrieve "query"` — any agent can shell out to it
- **Fallback** — read `_index.md` directly when PageIndex isn't configured

### Optional: qmd

[qmd](https://github.com/tobi/qmd) provides local BM25/vector hybrid search with LLM re-ranking, entirely on-device. Useful as a simpler alternative or complement to PageIndex for smaller wikis.

### Assistive Agent — Provider-Agnostic Wiki Maintenance

An optional lightweight LLM handles low-complexity wiki maintenance tasks, freeing the primary coding agent for cognitive work. The assistive agent operates through a **provider cascade**: local inference (Ollama), cloud API (OpenRouter), or any OpenAI-compatible endpoint.

**Default provider:** OpenRouter free tier (`openrouter/free`). Requires only an API key — no local hardware, no model downloads, no GPU.

**Alternative providers:**
- **Local** (Ollama / llama.cpp): $0.00, total privacy, requires 18–30GB RAM
- **OpenRouter budget** (Gemma 4 26B-A4B): $0.13/$0.40 per M tokens, higher rate limits
- **Self-hosted** (vLLM, TGI): custom endpoint, enterprise control
- **Any OpenAI-compatible API**: generic adapter

**Task routing by cognitive complexity:**

| Task | Complexity | Agent |
|------|-----------|-------|
| Frontmatter updates, `_log.md` entries | Low | Assistive |
| `_index.md` / `_Sidebar.md` regeneration | Low | Assistive |
| Link validation, cross-reference suggestions | Low–Medium | Assistive |
| Manifest updates, page state transitions | Low | Assistive |
| Simple module summaries, staleness checks | Medium | Assistive |
| Architecture synthesis, ADR creation | High | Primary |
| Contradiction detection, deep code analysis | High | Primary |
| Hash computation, path validation, orphan detection | N/A | Deterministic (no LLM) |

**Cascade behavior:** Try the cheapest configured tier first. On failure (unavailable, rate-limited, health check failed), fall through to the next tier. The primary agent is always the final fallback. High-complexity tasks skip directly to the primary agent regardless.

**Economics:** The assistive agent achieves ~83% reduction in wiki-related API costs by routing ~80% of wiki operations to free or near-free models. See `docs/reference/cloud-fallback-ref.md` for detailed token economics and `docs/reference/local-agent-ref.md` for hardware requirements and model benchmarks.

**Rate limit management:** For free-tier providers, the CLI tracks daily request counts and adapts batching strategy. Bulk operations (`plexium convert`) automatically escalate to the budget tier or queue work across multiple days.

**Privacy:** `privacyMode: strict` restricts routing to providers that don't log prompts for training. The `neverSend` config excludes sensitive content patterns from assistive agent routing.

### Symphony — Autonomous Execution Engine

[OpenAI Symphony](https://github.com/openai/symphony) provides an autonomous execution layer that complements Plexium's persistent knowledge layer. Plexium is the brain (compiled knowledge); Symphony is the nervous system (autonomous work dispatch).

**Integration tiers:**
- **Tier 1 (recommended, Milestone 5):** Inject Plexium's `_schema.md` directives into Symphony's `WORKFLOW.md`. Every Symphony-dispatched agent reads the wiki before working and updates it after. Zero architectural changes — one adapter plugin.
- **Tier 2 (Phase 3):** `plexium daemon` implements Symphony-style polling, workspace isolation, and retry/reconciliation using `manifest.json` as the state layer.
- **Tier 3 (future, if ever):** Full bidirectional integration where lint findings auto-generate tracker issues and Symphony dispatches wiki-update agents. Premature due to Symphony's pre-v1 status and Plexium's agent-neutral design.

**Key patterns adopted from Symphony:**
- `WORKFLOW.md` with YAML frontmatter for machine-readable config + Markdown body for agent instructions
- Per-issue isolated git worktrees with lifecycle hooks
- Exponential backoff retry policy
- Handoff states (`Human Review`) rather than binary done/not-done
- Restart recovery by reconciling against existing state (no external database)

**Runner interface:** The daemon's runner adapter is abstract — it invokes whatever agent the developer has configured (Claude, Codex, Gemini, or any future agent). Plexium does not couple to any specific agent runtime.

See `docs/reference/symphony-integration-ref.md` for the full strategic assessment, what-not-to-adopt analysis, and detailed design rationale.

### Agent Role Separation (Swarm Pattern)

For complex operations, the system supports delegating to specialized sub-agents:

| Role | Responsibility |
|------|---------------|
| **Coder** | Writes and modifies source code |
| **Explorer/Retriever** | Searches wiki + codebase via PageIndex, compiles context |
| **Documenter** | Updates wiki pages, maintains cross-references, runs lint |
| **Ingestor** | Processes raw sources into wiki pages |

In practice, a single agent session often handles all roles. For large operations (bulk `/convert`, periodic lint, batch ingest), role separation prevents context overflow and improves quality. Under daemon/orchestrated mode, roles are sequenced explicitly: **Retriever** (compile context via PageIndex) → **Coder** (implement changes) → **Documenter** (update wiki pages) → **Linter** (validate integrity).

## 17. Authentication Model

### Local Mode
- Developer provides a GitHub token or authenticated Git credential locally
- Token needs push access to both the main repo and `{repo}.wiki.git`
- Missing auth fails fast with clear remediation guidance
- Dry-run mode (`--dry-run`) does not require publish auth

### CI Mode
- GitHub Action uses a configured secret (`WIKI_PUSH_TOKEN`) or GitHub App token
- `GITHUB_TOKEN` (the default Actions token) has limited wiki push permissions — a dedicated PAT or App token is typically required
- Fine-grained PATs: must include "Contents: write" permission on the wiki repository
- Classic PATs: must include `repo` scope

### Permission Quirk
GitHub Wiki repos (`{repo}.wiki.git`) have separate permission semantics from the main repo. The default `GITHUB_TOKEN` in Actions can clone but often cannot push to the wiki repo. Documentation and error messages must be explicit about this. `plexium doctor` validates auth configuration.

## 18. Failure Modes & Safe Failure

### Invariants (Never Violated)

| Invariant | Description |
|-----------|-------------|
| **Never modify source code** | Wiki operations never touch files outside `.wiki/` and `.plexium/` |
| **Never delete unmanaged pages** | Pages with `ownership: human-authored` are never overwritten or removed by automation |
| **Never publish partial output** | If any step fails mid-update, roll back all wiki changes in the commit. No half-updated wikis. |
| **Never silently proceed on invalid config** | If `config.yml` or `_schema.md` is malformed, fail fast with actionable remediation |
| **Never commit references to nonexistent files** | `source-files` frontmatter and `[[wiki-links]]` are validated before commit |
| **Never remove cross-references silently** | Deletions of existing `[[links]]` must be logged in `_log.md` |

### Common Failure Cases & Handling

| Failure | Handling |
|---------|---------|
| Wiki not enabled on GitHub repo | `plexium doctor` detects and provides instructions to enable |
| Auth/token invalid or insufficient | Fail fast with specific remediation (which permissions are missing) |
| Duplicate page slug collision | Deterministic deduplication (append qualifier), log resolution |
| Malformed markdown source | Skip file, log warning in report, continue processing |
| Source file removed but mapping exists | Flag as stale in lint report, mark page for review |
| Sidebar links point to missing pages | Deterministic validation catches before publish |
| LLM generates hallucinated content | `confidence` frontmatter + `review-status` tracking + human review |
| Agent modifies `human-authored` page | Pre-commit hook rejects (deterministic check on frontmatter) |
| Manifest drift from actual wiki state | `plexium doctor` detects and offers `plexium manifest rebuild` |
| Concurrent agents update same page | See [Scaling Considerations](#20-scaling-considerations) |

### Fallback Behavior
- Stop publish on structural validation errors
- Allow dry-run artifact generation for debugging: `plexium sync --dry-run` outputs to `.plexium/output/`
- Produce actionable remediation output in all error messages
- Never exit silently — always emit a report

### Retry Semantics

For transient failures (LLM API errors, GitHub push rate limits, PageIndex unavailability), Plexium applies exponential backoff:

- **Default policy:** 3 attempts, 5s initial delay, 2× backoff multiplier, 60s max delay
- **Configurable** via `retry` block in `.plexium/config.yml`
- **Scope:** applies to all external API calls (LLM providers, GitHub API, issue trackers)
- **Daemon mode:** failed work items are released back to the queue after max retries, with the failure logged to `_log.md`

## 19. Greenfield vs. Brownfield Workflows

### Greenfield: Doc-Driven Development

In a new project, the agent follows an explicit **Plan → Execute → Compile** loop:

1. **Plan:** You prompt the agent with a feature request. It writes a proposed architecture into `.wiki/decisions/` and creates relevant module stubs *before writing any code*.
2. **Execute:** The agent writes the implementation.
3. **Compile:** The agent updates `.wiki/modules/` to reflect what the code *actually does* (not what was planned — reality wins). Updates `_index.md`, `_log.md`, `_Sidebar.md`, and cross-references.

The wiki grows organically from day one. By 50 modules, you have 50 module pages, dozens of cross-references, and a complete decision history. New team members (human or agent) orient themselves in minutes.

### Brownfield: Progressive Enrichment

1. Run `plexium convert` to bootstrap the wiki from existing docs and code structure
2. Review the conversion report — verify captured knowledge, note gaps
3. Fix highest-priority stubs first (the converter ranks them by estimated importance based on import frequency and file size)
4. As agents work on code changes, the mandatory schema ensures they update wiki pages for every touched module
5. Over 2–4 weeks of normal development, the wiki fills in organically from the points of active development outward
6. Run periodic `plexium lint --full` sessions to identify remaining gaps
7. Promote key pages from `unreviewed` to `human-verified` during sprint reviews

## 20. Scaling Considerations

### Small Projects (~10K LOC, 1–3 developers)
- `_index.md` is sufficient for navigation
- PageIndex not needed
- Single-file `manifest.json` works fine
- Advisory strictness is often sufficient — small teams self-enforce

### Medium Projects (~100K LOC, 5–15 developers)
- PageIndex becomes valuable for retrieval
- Full CI enforcement recommended
- Memento integration provides audit trail for distributed teams
- Weekly scheduled lint runs

### Large Projects / Monorepos (~500K+ LOC, 20+ developers)
- `_index.md` becomes insufficient → PageIndex required
- **Module-level sub-indexes:** each `modules/` subdirectory gets its own `_index.md`
- **CODEOWNERS-style governance:** `.plexium/owners.yml` maps wiki sections to responsible teams
- **Monorepo scoping:** Lefthook + Plexium scope hooks to specific packages, each with their own wiki subdirectory
- **Merge conflict mitigation:**
  - `_log.md` is append-only — conflicts trivially resolvable (keep both entries)
  - Module pages are scoped — different developers working on different modules rarely cause wiki conflicts
  - When pages do conflict: wiki Markdown conflicts are simpler than code conflicts, and agents can resolve them
  - For high-contention pages (`architecture/overview.md`): convention of appending `## Recent Changes` sections rather than rewriting the body, with periodic consolidation

### Concurrency Under Parallel Orchestration

Under daemon mode with `maxConcurrent > 1`, shared navigation files (`_index.md`, `_Sidebar.md`, `Home.md`, `_log.md`) become merge-conflict hotspots. The solution:

- **Agents update only content pages** (module, decision, concept pages) and append to per-run log files (`.plexium/workspaces/<ID>/run-log.md`)
- **`plexium compile`** deterministically regenerates all shared navigation files from page state — no LLM calls, no conflicts
- **`_log.md`** entries from parallel runs are merged by timestamp (append-only, chronologically sorted)
- This compilation step runs automatically after each daemon run and by CI on merge

In non-daemon mode (standard developer workflow), agents directly update navigation files as before — the concurrency problem only exists under parallel orchestration.

### Token Costs Under Orchestration

Parallel orchestration multiplies token consumption per unit time. Mitigations:
- **PageIndex becomes essential** — targeted retrieval costs far fewer tokens than each agent reading the full index
- **Cache wiki reads** across concurrent runs when the wiki hasn't changed between dispatches
- **Budget limits:** `daemon.tokenBudgetPerRun` and `daemon.tokenBudgetPerHour` config options, with the daemon pausing dispatch when limits are hit
- **Prioritize deterministic operations** — staleness detection, link validation, and manifest reconciliation require zero LLM tokens

### GitHub Wiki Submodule Under Concurrent Pushes

Concurrent pushes to `{repo}.wiki.git` will fail under parallel operation. Solutions:
- **Serialize wiki publishes** — `plexium compile` + `plexium publish` is a single-writer operation via lock file
- **Batch publishing** — collect changes and publish once per CI cycle or daemon poll interval
- **Sync-and-push model** may be the better default for daemon mode (see Open Design Question #6)

### Schema Migration
When wiki conventions change (new frontmatter field, section rename, structural change):
- `plexium migrate` applies schema changes across all existing pages
- `_schema.md` includes `schema-version: 1` — version is bumped on breaking changes
- Migration scripts live in `.plexium/migrations/` — similar to database migrations
- Migrations are deterministic and reversible where possible

### Schema Migrations During Active Orchestration

When `plexium migrate` bumps `schema-version` during active daemon operation:
- **Drain before migrate:** signal the daemon to stop claiming new work, wait for in-flight runs to complete (configurable timeout), apply migration, resume daemon
- **Version assertion:** each daemon run checks `schema-version` at start — abort and re-queue if it doesn't match the dispatched version
- **Non-breaking migrations** (adding optional frontmatter fields) don't require drain
- **Breaking migrations** (renaming sections, changing required fields) require drain + rebuild

## 21. Security & Trust

### Agent-Generated Content Risk
Agents can introduce hallucinated or subtly incorrect information. Mitigations:

| Mechanism | Purpose |
|-----------|---------|
| `review-status` frontmatter | Every agent-written page starts as `unreviewed`. Humans promote to `human-verified`. |
| `confidence` markers | Agents mark uncertain claims. Dataview surfaces these for review. |
| PR diff review | Wiki changes appear in PR diffs alongside code, subject to same review process. |
| Ownership model | `human-authored` pages cannot be modified by agents. |
| Deterministic validation | Structural correctness (links, paths, frontmatter) is verified without LLM. |

**Framing:** This is not worse than the status quo. Agents already write code that goes through review. Wiki pages should receive the same scrutiny — Plexium makes documentation *visible and reviewable* rather than absent.

### Sensitive Information
- `plexium init --private` creates `.plexium/sensitivity-rules.yml`
- The schema instructs agents: "Never include API keys, secrets, PII, or content marked as CONFIDENTIAL in wiki pages"
- `githubWiki.exclude` config prevents sensitive pages from publishing to the public wiki tab
- `.wiki/raw/` can be added to `.gitignore` for projects with sensitive source documents
- For open-source repos: the wiki can be a private submodule
- `plexium lint --security` scans for accidental secret exposure in wiki pages

### Privacy by Provider Tier

When using the assistive agent, data handling varies by provider:

| Provider | Data Handling | Suitable For |
|----------|--------------|-------------|
| **Local (Ollama)** | Never leaves the machine. Total privacy. | Proprietary codebases, regulated industries, air-gapped environments |
| **OpenRouter Free** | Provider-dependent. Some models log prompts for training. | Open-source projects, non-sensitive wiki content |
| **OpenRouter Paid** | Generally better data policies than free tier. | Most commercial projects |
| **Primary Agent** | Depends on provider (Anthropic, OpenAI, Google policies). | Whatever the team already trusts for code |

`privacyMode: strict` in config restricts routing to providers that don't log prompts for training. The `assistiveAgent.neverSend` patterns exclude sensitive content from assistive agent routing entirely.

### Token Cost & Latency
Mandating agents read `_index.md` + relevant pages adds tokens per task. Mitigations:

- **The index is small:** `_index.md` is one-liners, typically 2–5K tokens even for large projects
- **Targeted reads:** Agents fetch only relevant pages via PageIndex, not the entire wiki
- **The tradeoff is favorable:** Reading 3–4 wiki pages (5K tokens) is far cheaper than an agent spending 20+ tool calls grepping through source files to rediscover context
- **Trivial change exception:** The schema allows skipping full wiki updates for single-file, non-architectural changes

## 25. Open Design Questions

These remain unresolved and should be decided during implementation:

1. **Excerpt vs. synthesis:** Should generated pages preserve excerpts from source docs (quoting), or only synthesize summaries (paraphrasing)? Excerpts are more trustworthy but create duplication.

2. **One-to-many mapping:** Should source-to-wiki mapping allow one source file to generate multiple wiki pages in Phase 1? (e.g., a large `auth/` module producing separate pages for JWT, RBAC, sessions)

3. **Opinionated taxonomy:** Should page taxonomy be opinionated by default (Architecture/Modules/Decisions/Patterns/Concepts) or fully customizable from the start?
